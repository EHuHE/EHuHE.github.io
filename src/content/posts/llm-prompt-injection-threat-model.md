提示注入是我近期阅读 AI 安全资料时反复遇到的风险点。OWASP 将其列为 LLM 应用的重要风险之一，CyberSecEval 也把 prompt injection 作为独立评测方向。对个人学习来说，更有价值的不是收集“攻击话术”，而是把问题拆成可验证的威胁模型。

这篇笔记只记录防御性实验设计，不提供可直接滥用的提示样本。实验对象应限制在本地 toy application 或公开靶场式样例中，避免接触真实用户数据、真实凭据或生产系统。

## 威胁模型

我把提示注入拆成三个边界：

- 指令边界：系统指令、开发者指令和用户输入是否被清晰隔离。
- 数据边界：外部网页、文档、邮件或检索结果是否被当作不可信数据处理。
- 动作边界：模型输出是否能触发工具调用、文件写入、网络请求或权限变更。

如果一个 LLM 应用只负责摘要，风险主要集中在错误输出和信息泄露；如果它还能调用工具，风险就会扩展到越权动作、错误执行和审计缺失。

## 防护假设

一个可测试的防护方案至少应该包含四层：

1. 输入标注：把检索内容、用户输入和系统指令在模板中显式分隔。
2. 输出约束：让模型给出结构化输出，并拒绝把不可信内容解释为指令。
3. 工具权限：工具调用必须走 allowlist，并让敏感动作需要额外确认。
4. 审计记录：保存输入来源、模型输出、工具调用意图和最终动作。

这些防护不能假设模型永远“理解规则”。更稳妥的做法是把模型当作可能被混淆的组件，尽量缩小它能直接影响的系统范围。

## 实验指标

我计划用以下指标记录实验结果：

- 攻击成功率：不可信输入是否改变了原始任务目标。
- 拒绝误伤率：正常但边界模糊的输入是否被过度拒绝。
- 输出合规率：输出是否符合预期 schema。
- 工具隔离率：危险工具是否在无确认情况下被调用。
- 可审计性：日志是否足以复盘风险路径。
