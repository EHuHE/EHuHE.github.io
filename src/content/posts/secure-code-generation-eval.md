LLM 可以帮助生成代码，但安全代码生成不能只看“能不能运行”。CyberSecEval 把 secure code generation 作为评测方向，OWASP LLM 风险也提醒我们关注不安全输出处理和供应链问题。我的学习目标是设计一套小规模、可复现、偏防御的评测流程。

这类实验不需要真实漏洞利用环境。更合适的做法是准备一组安全需求明确的 toy tasks，用静态检查、单元测试和人工复核来判断模型输出是否引入常见风险。

## 评测对象

我会把任务控制在三类：

- 输入验证：检查用户输入是否被约束、解析和错误处理。
- 输出编码：检查 HTML、SQL、命令参数等上下文是否正确处理。
- 权限边界：检查文件、网络、令牌、环境变量等敏感资源是否被过度暴露。

这些任务可以映射到 CWE 或常见安全设计缺陷，但文章里只记录模式和检查点，不提供可用于攻击真实系统的操作步骤。

## 实验流程

一个安全代码生成评测可以分成五步：

1. 写出功能需求和安全需求，避免只给模型一个模糊任务。
2. 生成候选代码，保存模型、提示、时间和参数。
3. 运行单元测试，验证正常路径和错误路径。
4. 使用静态检查或规则扫描标记危险 API 和未处理输入。
5. 人工复核关键边界，记录“不安全但能运行”的情况。

实验结果不应该只写成分数。更有价值的是把失败模式归类，例如：遗漏输入长度限制、错误拼接命令、日志包含敏感字段、异常处理吞掉安全信号。

## 指标设计

我计划记录这些指标：

- 功能通过率：基础单元测试是否通过。
- 安全检查通过率：关键安全断言是否通过。
- 危险模式数量：是否出现不必要的动态执行、字符串拼接或过宽权限。
- 修复成本：需要多少修改才能达到安全需求。
- 解释质量：模型是否能解释安全取舍，而不是只给代码。
